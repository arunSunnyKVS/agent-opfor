# Prompt: Design the UI/UX for the Astra VS Code Extension

## Context

Astra is an open-source AI security testing tool (red teaming). It lets developers test their AI apps — chatbots, RAG pipelines, agents — against known vulnerabilities from the OWASP LLM Top 10 and OWASP Agentic AI Top 10.

The tool currently exists as:
- A **CLI** (`astra setup` → `astra run`)
- An **MCP server** (AI agents call `astra_setup` / `astra_run` tools)

We are now building a **VS Code extension** (Option 2 — directly uses `@astra/core`, no CLI subprocess). Your job is to design the complete UI/UX for this extension.

---

## What Astra Does (the workflow)

There are two steps:

### Step 1 — Setup
The user tells Astra about their target AI application and picks what to test. Astra uses an LLM to generate targeted attack prompts. Output: a `astra-prompts-*.json` file.

**Inputs collected during setup:**
- **Target name** — e.g. "Support Bot"
- **Target description** — what it does, what sensitive data it handles, dangerous operations it can perform, forbidden topics (more detail = better attacks)
- **Target type** — `http-endpoint` or `local-script`
- **Endpoint URL** — e.g. `http://localhost:4000/chat`
- **Request format** — `auto`, `openai` (messages array), or `json` ({prompt: "..."})
- **Target API key** — optional Bearer token for the endpoint
- **Evaluators** — which vulnerabilities to test for (pick a suite or individual evaluators)
- **Attack mode** — single-turn (one prompt → one response) or multi-turn (escalating conversation)
- **Turns** — if multi-turn, how many turns per attack (2–10)
- **Session ID field** — for multi-turn HTTP targets, which request body field carries the session ID
- **LLM provider** — Groq, OpenAI, Anthropic, Google (for generating attacks and judging responses)
- **Langfuse integration** (optional) — if the user has Langfuse set up, Astra can fetch production traces to make attack prompts much more realistic

### Step 2 — Run
Astra fires every generated attack prompt at the target, judges each response with an LLM (PASS/FAIL with score 0–10, evidence, reasoning), and produces a report.

**Output:**
- **Safety score** — overall % of attacks the target defended against
- **Per-evaluator results** — pass/fail counts and pass rate
- **Critical findings** — specific failed attacks with severity, evidence, reasoning
- **Full test log** — every prompt sent, every response received, every judge verdict
- **HTML report** — rich visual report saved to `.astra/reports/`

## Technical Constraints

- This is a **VS Code extension** — it lives in the VS Code sidebar and opens webview panels
- It imports `@astra/core` directly (TypeScript package) — no CLI subprocess
- API keys must be stored using VS Code's `SecretStorage` API (never in files)
- The extension auto-detects `astra.config.json` in the workspace root if present
- The extension can auto-detect Langfuse env vars from `.env` in the workspace
- Reports are saved to `.astra/reports/` in the workspace
- The existing HTML report (generated by `@astra/core`) can be loaded directly into a VS Code `WebviewPanel`
- The extension must work in both **VS Code** and **Cursor**

## Design Task

Design the complete UI/UX for this extension. Your output should include:

### 1. Information Architecture
- What views/panels exist?
- How does the user navigate between them?
- What lives in the sidebar activity bar vs webview panels vs notifications?

### 2. Setup Wizard Design
Design an intuitive, step-by-step setup flow inside a webview panel. The wizard should:
- Feel like a modern onboarding flow, not a form dump
- Auto-fill anything it can detect from the workspace (`astra.config.json`, `.env`, `package.json`, `README.md`)
- Clearly explain each field with inline help text — users may be new to red teaming
- Show a live preview of what evaluators will run before committing
- Make the Langfuse integration feel like a natural enhancement, not a complex optional
- Guide the user clearly to the "Run" step at the end

### 3. Run / Progress View
Design the live progress experience while attacks are running:
- How is progress shown? (per evaluator, per attack, overall)
- How does the user know when something is failing vs passing in real time?
- Can the user stop mid-run?

### 4. Results / Report View
Design how findings are presented inside VS Code (before/instead of opening the HTML file):
- The safety score should be immediately scannable
- Critical findings should surface urgently
- The user should be able to drill into individual attacks (prompt sent → response received → judge verdict)
- Multi-turn attacks have multiple turns — show this clearly

### 5. History / Scans View
Design a view showing past scans:
- How does the user browse previous runs?
- How do they compare two runs?

### 6. Settings / API Keys View
Design how the user manages:
- LLM provider and API key (stored in SecretStorage)
- Langfuse credentials (stored in SecretStorage)
- Default preferences (output directory, default provider, etc.)

### 7. Micro-interactions and Empty States
- What does the extension look like on first install (no scans yet)?
- What feedback does the user get when setup completes?
- What happens when a scan finds critical vulnerabilities?
- Error states (API key invalid, target unreachable, LLM error)

## Design Principles to Follow

- **Progressive disclosure** — don't show everything at once; reveal complexity only when needed
- **Confidence, not fear** — red teaming is a security practice, not an alarm. The UI should feel professional and empowering, not threatening
- **Zero config when possible** — auto-detect everything from the workspace; only ask for what's truly missing
- **Inline education** — users may not know what "prompt injection" means; explain it in context, not in a separate docs tab
- **Fast path** — an experienced user should be able to go from install to first scan result in under 2 minutes
- **VS Code native** — use VS Code's design language (colors, fonts, tree views, progress bars) so it feels like a native part of the IDE, not a web app bolted on

---

## Deliverables

1. **Screen-by-screen wireframe descriptions** — describe each screen in detail (layout, components, content). You don't need to produce images; detailed written descriptions of each view are the primary deliverable.

2. **User flow diagram** — map the complete journey from install → first scan → viewing results, including all decision points and alternate paths.

3. **Component inventory** — list every UI component needed (forms, lists, progress indicators, etc.) with their state variations (empty, loading, success, error).

4. **Copy guidelines** — the exact wording for key labels, headings, empty states, and error messages. Tone should be: professional, calm, developer-friendly, never alarmist.

5. **Key UX decisions** — for any design decision with real trade-offs (e.g. wizard vs single form, inline results vs separate panel), state the decision and explain why.
